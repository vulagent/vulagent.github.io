In this section, we describe the specific prompts used in our workflow, including those employed in the pruning, slicing, and LLM-based reasoning stages. Although the pruning and slicing components do not inherently depend on large language models and can be implemented through alternative techniques, we present here an engineering-oriented implementation that leverages LLMs for clarity and completeness.

Prompt for Pruning

Role: You are an expert in C/C++ integer overflow analysis. Task: Given the vulnerability context, determine whether the analyzed path is provably free from signed integer overflows for the given expression. Steps: 1. Identify the target expression and all relevant variables; infer each variable‚Äôs semantic role from naming and local/project context. 2. Analyze surrounding conditions, loops, and control flow to determine how these variables evolve and to derive feasible value ranges, incorporating global/configuration bounds where applicable. 3. Check whether any variable is compile-time constant (via macros/constant propagation) and, using the derived ranges, decide if a signed integer overflow is possible; if provably safe, mark the path as Pruned, otherwise Retained. Output: Provide a concise reasoning summary and clearly state whether the path contains a potential signed integer overflow.

Prompt for Slicing

Role: You are an expert in C/C++ integer overflow detection and program slicing. Task: Given the function body and a description of the potential integer overflow, extract only the statements that are semantically relevant to the variables and expressions contributing to the overflow, while preserving the essential execution semantics. Steps: 1. Identify the variables that participate in overflow-related expressions and determine their data dependencies using dataflow reasoning. 2. Retain only the statements that define, modify, or influence these variables, and remove unrelated code segments. 3. For the removed parts, insert concise natural-language summaries to maintain semantic completeness and readability. Expected Output: - A brief explanation of the slicing decisions and reasoning process. - The reduced and semantically coherent function body preserving overflow-relevant logic. - Short natural-language comments summarizing removed sections.

Prompt for LLM-based Reasoning

Role: You are an expert in C/C++ program analysis and semantic reasoning for vulnerability detection. Task: Given the current function and its surrounding program context, determine whether the function represents a user-reachable entry point and whether sufficient contextual information exists to construct a testcase that can reach the potentially vulnerable line. Steps: 1. Functionality Recognition: Interpret the semantics of the provided path to identify the enclosing function, infer its intended behavior, and analyze comments or descriptive identifiers that reveal its role in the program. 2. Key Variable Identification: Examine expressions related to integer operations within the path, identify the variables involved, and reason about their roles and value meanings in overflow-related computations. 3. Reflection and Decision-Making: Evaluate whether the current contextual knowledge is sufficient to construct a reachable testcase. If critical information is missing (for example, the input source of a variable is unknown), indicate that additional context should be retrieved. Otherwise, conclude whether a concrete testcase can be synthesized to exercise the target line. Output: A concise reasoning summary describing the inferred function semantics, the identified key variables, and a final judgment on whether the function is user-reachable and ready for testcase synthesis.
